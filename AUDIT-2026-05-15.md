## Executive Summary

LlevaLleva is a **Next.js 16 / React 19 / Supabase** general classifieds marketplace for Colombia (and aspirationally Venezuela). The codebase is substantially complete: 16 categories, full auth loop, listing publish/edit/delete, WhatsApp contact, transaction + rating system, a Gemini-powered price checker, Google AdSense with Consent Mode v2, and a solid SEO foundation.

**Score: 7.0 / 10**

The site is structurally sound and nearly launch-ready. The main blockers are:
- `GEMINI_API_KEY` is not in `.env.local` and not documented — the `/cuanto-vale` price tool silently 500-errors in production unless this is set in Netlify env
- Personal Gmail exposed in `consent-banner.js` served to all users
- Three AdBanner instances use `slot=""` (empty) — direct revenue/policy risk
- Duplicate WebSite JSON-LD injected on homepage (both `layout.tsx` and `page.tsx`)
- Conflicting `robots.txt` files (three copies, two wrong)
- Stale 2024 financial data presented as current fact in a public-info feature

## P0 — Critical / Launch Blockers

### P0-1: `GEMINI_API_KEY` not set — `/cuanto-vale` silently fails in production

**File:** `netlify/functions/price-check.mjs:50–58`, `app/cuanto-vale/PriceChecker.tsx:74–79`  
**Detail:** The Netlify function reads `process.env.GEMINI_API_KEY` and returns HTTP 500 if unset. `PriceChecker.tsx` catches the error and shows "Error de conexión" — which is opaque to the user. `.env.local` does not contain `GEMINI_API_KEY`. `.env.local.example` does not document it. The README contains no mention of it.  
**Fix:** Add `GEMINI_API_KEY=<your-key>` to Netlify environment variables (Settings → Environment Variables). Add it to `.env.local.example` with a comment. Update README with a "Required env vars" section.

---

### P0-2: `/api/price-check` is a Netlify Function, NOT a Next.js API route

**File:** `app/cuanto-vale/PriceChecker.tsx:75`, `netlify/functions/price-check.mjs:197–199`  
**Detail:** `PriceChecker.tsx` calls `fetch('/api/price-check', ...)`. There is no `app/api/price-check/route.ts` — the `app/api/` directory does not exist at all. The function lives in `netlify/functions/price-check.mjs` and uses `export const config = { path: '/api/price-check' }` (Netlify v2 path rewriting). This works correctly on Netlify production but fails completely in:
- `npm run dev` (local development)
- Any non-Netlify environment (CI, Docker, Vercel)

**Fix (Option A — recommended):** Create `app/api/price-check/route.ts` as a proper Next.js route handler that contains the same Gemini logic. Remove `netlify/functions/price-check.mjs` to avoid duplication. This works everywhere.  
**Fix (Option B — minimal):** Keep the Netlify function, add a local dev proxy in `netlify.toml` or `next.config.ts` to redirect `/api/price-check` to the function at `localhost:8888`. This still breaks non-Netlify envs.

---

### P0-3: Duplicate WebSite JSON-LD schema on every homepage load

**File:** `app/layout.tsx:100–113` AND `app/page.tsx:15–33`  
**Detail:** `layout.tsx` injects a `WebSite` JSON-LD schema block globally (affects every page). `page.tsx` (the homepage) injects an identical second `WebSite` JSON-LD block. Google Search Console flags duplicate structured data and may suppress rich results sitewide.  
**Fix:** Remove the `websiteJsonLd` constant and its `<script>` block from `app/page.tsx` entirely. The one in `layout.tsx` is already applied to all pages including the homepage.

---

### P0-4: Conflicting `robots.txt` — three copies, two incorrect

**Files:** `robots.txt` (repo root), `public/robots.txt`, `app/robots.ts`  
**Detail:**
- `app/robots.ts` — the authoritative Next.js dynamic route. Correctly disallows `/auth/` and `/dashboard/`, and blocks all major AI bots (GPTBot, ChatGPT-User, Google-Extended, CCBot, anthropic-ai, Claude-Web, ClaudeBot, etc.). **This is the one served in production.**
- `public/robots.txt` — static file, minimal (`Allow: /` only, no auth/dashboard block, no AI-bot blocks). **Overridden at runtime by `app/robots.ts` but confuses editors and some static hosting environments.** Also has a BOM character at line 1 (UTF-8 BOM).
- `robots.txt` (repo root) — identical to the minimal `public/robots.txt`. **Served from root when site is not a Next.js deployment (e.g., `netlify dev` static mode). Should not exist.**

**Fix:** Delete `robots.txt` (repo root) and `public/robots.txt`. The dynamic `app/robots.ts` is correct and sufficient.

## P1 — High Priority

### P1-1: Personal Gmail exposed in `public/consent-banner.js`

**File:** `public/consent-banner.js:30`  
**Detail:** The comment block reads `* - Email : ajsgallie@gmail.com` with a note "PENDING: migrate to custom domain email once available". This comment is served to every user as a public JS file. Under Ley 1581 de 2012 (Colombia) data controller disclosure requirements, the data controller contact should be a business email, not a personal Gmail.  
**Fix:** Replace `ajsgallie@gmail.com` with `hola@lleva-lleva.com`. The comment is internal developer documentation — if it must stay, move it to a `// ...` comment that minifiers will strip, or remove the email line entirely since the NIT and address are sufficient for legal disclosure.

---

### P1-2: Twitter/X card handle inconsistency across pages

**File:** `app/layout.tsx:59–63` vs `app/cuanto-vale/page.tsx:33–36`  
**Detail:** The root layout sets `twitter.site = '@MaiaGroupCO'` and `creator = '@llevalleva'`. The `/cuanto-vale` page sets both `site` and `creator` to `'@llevalleva'`. This means the company handle shown in Twitter cards changes depending on which page is shared.  
**Fix:** Decide on one handle per field. If `@llevalleva` is the brand account, update `app/layout.tsx` to use `@llevalleva` for `site`. If `@MaiaGroupCO` is intended for the organization, document that decision.

---

### P1-3: Venezuela in JSON-LD and JSON-LD description, absent from all visible copy

**File:** `app/layout.tsx:106–107`  
**Detail:** The `WebSite` JSON-LD sets `inLanguage: ["es-CO", "es-VE"]` and description "Clasificados generales en Colombia y Venezuela". No page, heading, or visible copy mentions Venezuela as a target market. The homepage hero says "toda Colombia". This mismatch confuses search engines and may cause irrelevant indexing.  
**Fix:** If Venezuela is not currently an active market, remove `"es-VE"` and "Venezuela" from the JSON-LD. If it is planned, add at least a brief mention in the homepage copy and About page.

---

### P1-4: Three AdBanner instances use `slot=""` (empty)

**Files:** `app/page.tsx:169` (hero→category gap), `app/page.tsx:266` (public info section), `app/categorias/[slug]/page.tsx` (top of category page)  
**Detail:** `AdBanner` receives `slot=""`. The component converts this to `adSlot = 'auto'`. When AdSense receives a slot ID of `'auto'`, it enters auto-ad mode for that `<ins>` element — behavior that is not predictable and may not generate impressions or revenue. Two of the three banners above are in high-visibility positions.  
**Fix:** Create dedicated AdSense ad units in your AdSense account and add their slot IDs to Netlify env vars (`NEXT_PUBLIC_ADSENSE_SLOT_HOMEPAGE_TOP`, `NEXT_PUBLIC_ADSENSE_SLOT_INFO_SECTION`, `NEXT_PUBLIC_ADSENSE_SLOT_CATEGORY_TOP`). Pass them as `slot={process.env.NEXT_PUBLIC_ADSENSE_SLOT_*}` on each AdBanner. The two homepage banners that already use `process.env.NEXT_PUBLIC_ADSENSE_SLOT_HOMEPAGE_BANNER` and `NEXT_PUBLIC_ADSENSE_SLOT_HOMEPAGE_FEED` are correctly wired — just set those env vars in Netlify.

### P1-5: Stale 2024 financial data in `/info/precios-referencia`

**File:** `app/info/[category]/[location]/page.tsx:33–36`  
**Detail:** The page hardcodes "Salario Mínimo 2024: $1.300.000 COP" and "UVT 2024: $47.065 COP". The 2026 values are SMMLV ~$1.423.500 and UVT ~$49.799. The legal pages on this site are dated April 2026. Showing two-year-old wage data as reference information undermines trust.  
**Fix:** Update to 2026 values with a "Datos 2026" label. Better: replace the hardcoded numbers with a direct link to `www.mintrabajo.gov.co` for SMMLV and `www.dian.gov.co` for UVT, and display only the year with a "verify with official source" caveat.

### P1-6: "Próximamente" placeholder content live in production

**File:** `app/info/[category]/[location]/page.tsx:52–55, 63–66`  
**Detail:** The `noticias` and `alertas` info-page categories render content like "Próximamente: integración con fuentes de noticias locales verificadas." These are accessible from the homepage Información Pública section. Showing "coming soon" content as live pages looks unfinished and may draw Google's "thin content" penalty.  
**Fix (A):** Remove `noticias` and `alertas` from `PUBLIC_INFO` in `app/page.tsx` until the content is real.  
**Fix (B):** Add `noindex` to the metadata for those specific category/location combinations.

### P1-7: Root metadata description undersells the platform's breadth

**Files:** `app/layout.tsx:17`, `app/page.tsx:10`  
**Detail:** Both the default metadata and homepage metadata say "Vehículos, inmuebles, tecnología, náutico y más" — listing 4 of 16 categories. The description is used for Google's snippet. Users searching for empleo, mascotas, agro, servicios, or moda will not see these terms in search results.  
**Fix:** Expand to: "El clasificado colombiano. Compra, vende y conecta — vehículos, inmuebles, tecnología, empleo, servicios, náutico, mascotas, agro y mucho más en lleva-lleva.com."

## What Is Working Well

- **WhatsApp routing is correct.** `lib/utils.ts` hardcodes `WHATSAPP_BOT_NUMBER = '19034598763'`. Bot-seller listings route to the Maia bot. Real sellers get direct contact. The global float button links to `wa.me/19034598763`. No wrong numbers anywhere.
- **NIT is correct everywhere.** All references use `901.862.977-7` (the `-7` suffix). No `-1` errors found.
- **Domain is correct.** `lleva-lleva.com` is used consistently in all canonical URLs, OG tags, JSON-LD, sitemap, and robots. The old `llevalleva.co` domain only appears in a migration file that explicitly fixes it — the correct pattern.
- **Security headers are comprehensive.** `next.config.ts` sets X-Frame-Options, X-Content-Type-Options, Referrer-Policy, HSTS, Permissions-Policy, and a detailed CSP covering Supabase, Google Ads, Google Analytics, and AdSense.
- **Consent Mode v2 is properly implemented.** `consent-banner.js` fires `gtag('consent', 'default', {...denied})` before any tracking scripts load. AdBanner checks localStorage before rendering. GA4 is conditionally loaded only when the env var is set.
- **Canonical URLs on every page.** All route-specific metadata includes `alternates.canonical`. No missing canonicals found.
- **Structured data is thorough.** Organization, WebSite, ContactPage, AboutPage JSON-LD implemented. `SearchAction` on WebSite enables Google Sitelinks Search Box.
- **RLS and auth are server-side.** All protected routes use `createClient()` (server) and check auth via Supabase SSR. No client-only auth guards that could be bypassed.
- **All 16 categories are in sitemap.** `app/sitemap.ts` hardcodes all 16 slugs as a fallback even if the DB is unavailable at build time.
- **`app/robots.ts` correctly blocks AI scrapers** and protects `/auth/` and `/dashboard/` paths.
- **No personal emails or wrong phone numbers in any public-facing component.** `contact-banner.js` is the only exception (P1-1).
- **No exposed secrets in git.** `.env.local` is gitignored and not tracked. The `SUPABASE_SERVICE_ROLE_KEY` format (`sb_secret_...`) is the new Supabase project-scoped token format, not a JWT — this is correct and expected.
